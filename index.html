What's actually happening when a query runs

Write a query with SELECT at the top and WHERE in the middle — but that's not the order SQLite (or any SQL engine) actually executes it in. The engine works through clauses in a fixed logical sequence, rebuilding and shrinking the row set at every step. The diagram above runs your exact query through sql.js — real SQLite compiled to WebAssembly — and shows you the true intermediate result after each stage, not a scripted animation.

Try editing the query above. The pipeline re-executes live: every row count, every discarded row, every re-sorted chip reflects an actual query sql.js just ran against the in-browser database — nothing here is hand-authored per example.

FROM & JOIN

Execution starts here, not at SELECT. The engine loads the base table(s) and, if a JOIN is present, matches rows between tables on the join condition before anything else happens. This is why a query can reference a joined table's column in its WHERE clause — by the time filtering starts, the join has already produced one combined row set.

WHERE

Every row from the FROM/JOIN stage is tested against the condition one at a time. Rows that evaluate true continue forward; rows that evaluate false are discarded immediately — permanently, for the rest of this query. Aggregate functions can't appear in WHERE for exactly this reason: aggregates don't exist yet at this point in execution.

GROUP BY

Surviving rows are bucketed into groups that share the same value(s) in the GROUP BY column(s). Once grouped, individual rows stop being individually addressable — anything downstream operates on the group as a unit, which is why SELECT can only reference grouped columns or aggregate functions like COUNT(), SUM(), or AVG() from here on.

HAVING

HAVING is WHERE's counterpart for groups instead of rows — it filters out entire groups based on a condition, commonly one involving an aggregate (e.g. HAVING COUNT(*) > 3). This is the reason HAVING can filter on aggregates when WHERE cannot: groups (and their aggregate values) don't exist until after GROUP BY has run.

SELECT & DISTINCT

Only now does the engine evaluate the SELECT list — projecting down to the specific columns or expressions you asked for, discarding everything else. If DISTINCT is present, duplicate resulting rows are collapsed into one at this same stage.

ORDER BY

With the final row shape decided, the engine sorts the result set. This runs after SELECT, which is why you can ORDER BY a column alias that was only just defined in the SELECT list.

LIMIT / OFFSET

Finally, the engine trims the sorted result down to the requested window of rows. Because this is the very last step, LIMIT always applies to the final, filtered, grouped, sorted output — never to any intermediate stage.
